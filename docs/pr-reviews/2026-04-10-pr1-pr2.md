## PR #1 — `implement factory and smart account deployment`
**Author:** Levi (leevigh)
**URL:** https://github.com/3000-Labs/latch/pull/1
**Additions:** 968 | **Deletions:** 182
**Status:** Request Changes

### What it does
- Adds a new `/api/smart-account/factory` route that deploys smart accounts via a factory contract
- Fixes a crash-at-boot issue by moving env var validation from module-level into lazy `getConfig()` getters (good fix — Next.js edge runtime would throw on module init)
- Updates the auth payload structure to use the new `AuthPayload` named struct with `context_rule_ids` + `signers`

### Issues

#### 1. In-memory cache is a serverless footgun
**File:** `app/api/smart-account/factory/route.ts`
```ts
const deployedAccounts: Map<string, {...}> = new Map();
```
This is module-level state. On Vercel/serverless, each cold start gets a fresh map. The `POST` handler checks the cache before calling `fundAccountIfNeeded`, so on a cold instance it'll try to fund an already-funded account unnecessarily. The GET route has the same issue — cache miss leads to an RPC roundtrip every time.
- **Action:** Fine for demo, but add a comment acknowledging this limitation.

#### 2. `setup_bundler.js` logs `P` + pubkey instead of pubkey (likely typo)
**File:** `setup_bundler.js`
```ts
console.log(`P${kp.publicKey()}`);
```
The `P` prefix is likely accidental — G-addresses already start with `G`.
- **Action:** Remove the `P` prefix.

#### 3. `fundAccountIfNeeded` silently swallows errors
**File:** `app/api/smart-account/factory/route.ts`
```ts
} catch (err) {}
```
If the Horizon check throws for a non-404 reason (network error, etc.), it silently falls through to Friendbot.
- **Action:** Fine for demo, but worth a comment explaining the intent.

#### 4. Salt versioning — looks good
The `SMART_ACCOUNT_VERSION = "factory-v2"` approach is clean and intentional. No action needed.

### Verdict
Close to mergeable. Fix items 1 and 2 (item 3 is optional but nice to have), then approve.

## PR #2 — `feat: add Stellar demo page and update project dependencies`
**Author:** Michael (kcmikee)
**URL:** https://github.com/3000-Labs/latch/pull/2
**Additions:** 14,561 | **Deletions:** 3,845
**Status:** Request Changes

### What it does
- Adds `/demo-stellar` — a standalone page for connecting standard Stellar wallets (Freighter, Lobstr, etc.) and incrementing a counter contract
- Uses `@creit.tech/stellar-wallets-kit` for multi-wallet support
- Adds a nav button on the Hero for the new demo

### Issues

#### 1. Dependency conflict in `package.json` (blocks merge)
**File:** `package.json`
```json
"@stellar/freighter-api": "^2.0.0",
"@creit.tech/stellar-wallets-kit": "^2.1.0"  // internally requires freighter-api ^6.0.1
```
The wallets kit requires `freighter-api ^6.0.1` but `package.json` pins it at `^2.0.0` directly. npm will resolve to 2.x and likely break the kit.
- **Action:** Remove the direct `@stellar/freighter-api` and `@lobstrco/signer-extension-api` entries from `package.json` — the wallets kit manages those as its own dependencies.

#### 2. `COUNTER_ADDRESS` hardcoded in two places
**File:** `app/demo-stellar/page.tsx`
```ts
// inside runDemo()
const COUNTER_ADDRESS = "CBRCNPTZ7YPP5BCGF42QSUWPYZQW6OJDPNQ4HDEYO7VI5Z6AVWWNEZ2U";

// also defined again outside runDemo as a standalone const
const COUNTER_ADDRESS = "CBRCNPTZ7YPP5BCGF42QSUWPYZQW6OJDPNQ4HDEYO7VI5Z6AVWWNEZ2U";
```
Should use `process.env.NEXT_PUBLIC_COUNTER_ADDRESS` to stay consistent with the rest of the app, and defined only once.
- **Action:** Replace both with `process.env.NEXT_PUBLIC_COUNTER_ADDRESS`.

#### 3. `stellar-sdk` major version bump (`^13.3.0` → `^14.5.0`)
The lockfile diff is large. The `rpc.Api` and `TransactionBuilder` APIs shifted between v13 and v14.
- **Action:** Confirm no breaking changes affect the existing smart account routes before merging.

#### 4. Minor: `catch (err: any)` type usage
Several `catch (err: any)` blocks in the page component. Not a blocker for a demo.

### Verdict
Needs the dependency conflict resolved (item 1) before merging — that's a real bug. Items 2 and 3 should also be addressed.
